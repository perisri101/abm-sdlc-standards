ABM IT SDLC Guardrails & Enforcement Playbook

Defines how SDLC controls are enforced at pipeline, process, and governance layers. Reference this when configuring CI/CD pipelines and branch protection.
1 Three Enforcement Layers

Every SDLC control is enforced through one or more of three layers. Automated pipeline gates are the first line — they cannot be bypassed without explicit approval. Human process gates add judgment for complex decisions. Governance provides structural accountability and course correction.

Layer 1 — Pipeline
Automated · Blocks Merges

Automated checks run in CI/CD on every pull request and commit. Failures block promotion — no human can override without an approved exception on record.

  • Build compilation checks
  • Unit test pass + coverage gate
  • SAST (Snyk Code / SonarQube)
  • SCA — dependency CVE scan
  • Secrets scan (Gitleaks)
  • Code complexity check
  • DAST (pre-prod stage)
  • Performance regression gate
Layer 2 — Process
Human Gates · Blocks Promotion

Human checkpoints that require deliberate review before a change moves to the next environment. Enforced via branch protection rules and required approvals.

  • PR review (2 approvers + TM)
  • Security Assessment sign-off
  • ADR review for arch decisions
  • UAT sign-off by product owner
  • Production approval gate (TM)
  • Change ticket linked to PR
  • Rollback plan documented
Layer 3 — Governance
Structural · Periodic Reviews

Structural controls and periodic audits that ensure the SDLC remains effective and compliant over time. Run by SDLC Gatekeeper and TMs quarterly.

  • Child SDLC attestation (annual)
  • SDLC health metrics review (quarterly)
  • Audit evidence collection (ISO)
  • Exception log review (monthly)
  • Pipeline gate effectiveness review
  • Tech debt ratio tracking
  • Post-incident SDLC review

3 CI/CD Pipeline Gates Specification

Every gate listed as Required must be present in all team CI/CD pipelines. Gates may not be skipped or soft-failed without an approved exception on record. Configure failure actions exactly as specified.

Gate Name What It Checks Required? Tool Failure Action Configuration Notes
Build Gate Compiles cleanly with no errors or unresolved warnings set to error-level Required GitHub Actions / ADO Block merge. Build must be green before any other gate runs. Treat warnings as errors (warningsAsErrors=true). Fail-fast on first build error.
Unit Tests 100% of unit tests pass. Code coverage ≥ 70% on new/changed code Required xUnit / Jest / pytest Block merge. Coverage drop below 70% on changed lines is a build break. Coverage measured on changed files only (delta coverage). Configure coverage threshold in pipeline YAML. Report published as PR comment.
SAST Gate Static analysis for security vulnerabilities and code quality in source code Required Snyk Code / SonarQube Zero High or Critical findings permitted. Block merge. Medium findings create a warning comment but do not block. All Critical/High findings must be remediated or have an accepted exception before merge.
SCA Gate Third-party dependency CVE scan. Identifies vulnerable packages Required Snyk Open Source Zero Critical or High CVEs in direct dependencies. Block merge. Run on every PR. Snyk monitor enabled for continuous monitoring. Automated PR created when fix available. Critical CVEs in transitive deps reviewed within 5 days.
Secrets Scan Detects hardcoded credentials, API keys, tokens, connection strings in source and commit history Required Gitleaks / TruffleHog Zero tolerance. Single finding blocks merge. Treat as a security incident. Scan full commit history on first run. Scan delta on subsequent PRs. Pre-commit hook also recommended for local dev. Any historical finding must be rotated immediately.
Complexity Check Cyclomatic complexity per function / method must not exceed threshold Required SonarQube / ESLint Cyclomatic complexity >10 per function blocks merge. New functions only. Apply to new and modified functions. Legacy code exempted but flagged as tech debt in backlog. Set SonarQube Quality Gate threshold = 10.
DAST Gate Dynamic security scan of running application — auth, injection, OWASP Top 10 Required StackHawk Zero Critical or High findings block UAT promotion. Runs at pre-prod stage. Run against deployed QA or UAT environment. Not on feature branches. Configure StackHawk with app-specific auth flow. Results reviewed by Security/TM before UAT sign-off.
Performance Gate Load and performance regression vs. baseline. Validates response time SLAs under expected concurrency Recommended k6 / Azure Load Testing P95 response time regression >20% over baseline blocks UAT promotion. Required for APIs handling >100 concurrent users or processing sensitive transactions. Baseline established at first deployment and versioned. TM may waive with documented justification.

5 Zero-Tolerance Control Matrix

The following six controls admit no exceptions without an approved, time-bound exception on record signed by the SDLC Gatekeeper. Violations are treated as security incidents.

6 Exception Process

All exceptions to SDLC controls must be formally requested and approved. Verbal approvals are not valid. Exceptions are time-bound and must be reviewed at expiry. Zero-tolerance control exceptions require SDLC Gatekeeper sign-off.

  1. 1
    Identify the Control to Be Excepted

    Document which specific SDLC control or pipeline gate requires an exception, including the control ID or section reference from the master SDLC or this guardrails document.

  2. 2
    Document Business Justification

    Complete the Exception Request Form below. The justification must explain why compliance is not feasible, what the business impact of non-compliance is, and what compensating controls are in place during the exception period.

  3. 3
    Technology Manager Approval

    The TM reviews the request and either approves (with conditions), rejects, or escalates. TM approval is required for all exceptions. TM cannot approve exceptions to zero-tolerance controls — those require Gatekeeper approval.

  4. 4
    SDLC Gatekeeper Approval (for Zero-Tolerance Controls)

    Exceptions to zero-tolerance controls must be reviewed and approved by Srini Peri. The Gatekeeper may add conditions, require additional compensating controls, or reject the exception entirely.

  5. 5
    Log and Track the Exception

    The approved exception is logged in the team's exception register in the SDLC repository. Exception ID assigned. Expiry date set (maximum 90 days). Reminder added to team calendar.

  6. 6
    Review at Expiry

    At the expiry date: either remediate (control implemented and exception closed), renew (re-submit with updated justification), or escalate (if systemic issue, escalate to SDLC Gatekeeper for structural remedy).

8 SDLC Health Metrics — KPI Dashboard

Technology Managers must track and report these KPIs quarterly to the SDLC Gatekeeper. Metrics are reviewed in the quarterly SDLC governance meeting. Significant negative trends must have a documented remediation plan.

Sprint Completion Rate
≥ 80%
% of committed story points delivered per sprint. Target: 80%+. Trend more important than single quarter.
Test Coverage Trend
≥ 70%
Unit test coverage on changed/new code. Must not decline quarter-over-quarter. Source: SonarQube.
SAST Finding Trend
0 Critical/High
Open Critical and High SAST findings at end of quarter. Target: zero. Trending down required. Source: Snyk/SonarQube.
Deployment Frequency
≥ 2×/month
Deployments to production per month. DORA metric. High-performing = multiple per day. Track trend. Source: CI/CD logs.
P1 Incident Rate
≤ 1/quarter
P1 production incidents per quarter. Target: <1. Each P1 must have a completed RCA. Source: Incident management system.
MTTR (Mean Time to Recover)
< 4 hours
Average time from incident detection to service restoration for P1/P2 incidents. DORA metric. Source: Incident log.
Tech Debt Ratio
≤ 5%
SonarQube tech debt ratio (remediation effort vs. total code cost). Must not grow QoQ without documented plan. Source: SonarQube.
PR Cycle Time
< 24 hours
Average time from PR opened to merged (excluding intentional hold). High cycle time indicates bottlenecks in review process. Source: GitHub/ADO.
